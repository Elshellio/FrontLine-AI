The client-facing procedure.

This is the practical sequence used to keep a project under control from first request through to live acceptance.

01

Define the change

We clarify what is being changed, why it matters, what outcome is expected and what should not be touched.

02

Inspect the current state

Before editing, we check the current page, workflow, form, API, design or behaviour so the change is based on reality.

03

Make one controlled change

Work is kept narrow. A design change is not mixed with unrelated backend, content, server or data changes unless agreed.

04

Review what changed

The actual difference is checked so accidental removals, broad rewrites or unrelated edits can be caught early.

05

Test the result

The changed area is tested in the browser, form, workflow or live route depending on what the change affects.

06

Accept and record

Once the change is approved, it becomes part of the controlled build history so future work starts from a stable point.

Decision lights.

Every change should be easy to classify. If the result is unclear, we pause rather than pushing uncertainty into the live system.

GreenThe change is understood, tested and accepted. It can move into the controlled build history.
AmberThe change works, but needs review, copy adjustment, design polish or a follow-up decision.
RedThe change is unclear, too broad, breaks expected behaviour or affects something outside scope.
