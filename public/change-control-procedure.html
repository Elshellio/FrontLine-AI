The non-negotiable rules

These rules keep LawFlow recoverable and stop broad accidental edits from becoming permanent.

✓

One change at a time

Each fix or feature must be small enough to inspect, test, review and roll back cleanly.

⌁

Inspect before editing

Never guess the file structure. Inspect the current page, route, backend block or Git state first.

Standard change control procedure

Use this sequence for Codex, ChatGPT-generated scripts or direct server changes.

Start clean

Before editing anything, confirm the working tree is clean. If Git is already dirty, stop and understand why.

Command
cd /opt/lawflow
git status
git log --oneline -3

Inspect the target

Find the exact file, route, UI block or backend function being changed. Codex should inspect first before writing.

Examples
grep -n "target text" /opt/lawflow/public/page.html
grep -n "route name" /opt/lawflow/api/server.js
node --check /opt/lawflow/api/server.js

Make one narrow change

Change only the intended target. Do not combine unrelated UI, backend, database, nginx or styling changes in one pass.

Rule
One change = one reason = one diff = one review = one commit.

Review the Git diff

Look at what actually changed. This catches accidental broad replacements, wrong files and generated clutter.

Commands
cd /opt/lawflow
git status
git diff --stat
git diff -- path/to/changed-file

ChatGPT review gate

Before committing, paste the diff summary and target diff into ChatGPT. Ask for a scope/risk/test review.

Minimum review pack
git status
git diff --stat
git diff -- path/to/changed-file

Test the changed area

Use the browser, curl, endpoint runner or live workflow test depending on what changed. Backend edits must pass syntax check.

Common tests
node --check /opt/lawflow/api/server.js
curl -I https://hub.missedcallsrecovered.co.uk/legal/change-control-procedure.html
pm2 status

Commit locally

Only commit after the diff has been reviewed and the relevant test has passed.

Commands
cd /opt/lawflow
git add path/to/changed-file
git commit -m "Clear description of the change"
git status

Push to GitHub

GitHub is the private off-server backup and review trail. It only updates when you push.

Command
cd /opt/lawflow
git push

ChatGPT review before commit

This is the quality gate between “a file changed” and “the change is accepted”. Use it before every commit, especially after Codex edits.

Prompt to paste into ChatGPT
Please review this LawFlow Git diff before I commit.

Goal of the change:
[describe the intended change in one sentence]

I need you to check:
1. Did the diff only touch the intended target area?
2. Are there accidental broad replacements or unrelated file changes?
3. Is there any obvious syntax, routing, UI, data or security risk?
4. What exact test should I run before commit?
5. Is this safe to commit, or should I stop?

Here is the output:

[PASTE git status]
[PASTE git diff --stat]
[PASTE git diff for the changed file]

Decision lights

Use this simple model when deciding whether to commit and push.

GreenDiff is narrow, test passes, ChatGPT review has no blockers. Commit and push.
AmberDiff is larger than expected or test coverage is weak. Review again before commit.
RedUnexpected files changed, syntax failed, or behaviour is unclear. Do not commit.
RollbackIf uncommitted, restore the file. If committed, use a revert commit.
CheckpointAfter commit and push, GitHub becomes the latest off-server recovery point.

The rule: no mystery changes.

Every LawFlow change should have a reason, a visible diff, a ChatGPT review, a test result, a local commit and a GitHub backup.
